Spring Boot实战派

发表于 2020-10-23 | 分类于 微服务

Spring Boot基础

Spring Boot设计初衷是解决Spring各版本配置过程工作过于繁重的问题，简化初始搭建流程，降低开发难度，使得开发人员只需要专注应用程序的功能和业务逻辑实现，而不用在配置上花费太多时间。

Resource和Autowired区别

Resource：继承于javax.annotation.Resource，默认是按照byName的方式来进行匹配

Autowired：Spring注解类，默认是按照byType的方式来进行匹配。

以上2个注解都可以通过byName和byType两个方式来处理的。

参考连接 https://www.cnblogs.com/think-in-java/p/5474740.html

RequestMapping

如果用在类上，则表示类中的所有响应请求方法都是以该地址作为父路径的，该注解有6个属性：

  1. Params：指定Request中必须包含某些参数值，才让该方法处理
  2. Headers:指定Request中必须包含指定某些指定的Header值，才能让某方法处理该请求
  3. Value：指定请求的实际地址，指定地址可以是URL Template模式
  4. Method：指定请求的Method类型，如GET，POST，PUT，DELETE等
  5. Consumes:指定处理请求的提交内容Content-Type，如”application/json,text/html”
  6. Produces:指定返回的内容类型。只有当Request请求头中的Accept类型中包含该指定类型时才返回。

Spring工作流程

  1. 客户端发出的请求由Tomcat接收，然后Tomcat将请求转发给DispatcherServlet处理
  2. DispatcherServlet匹配控制器中配置的映射路径，进行下一步处理
  3. ViewResolver将ModelAndView或Exception解析成View。然后View会调用render()方法，并根据ModelAndView中的数据渲染出页面。

Thymeleaf解决了前端开发人员要和后端开发人员配置一样环境的尴尬和低效。它通过属性进行模板渲染，不需要引入不能被浏览器识别的新的标签。页面直接作为HTML文件，用浏览器打开页面可看到最终的结果，可以降低前后端人员的沟通成本。

Webflow

对于WebFlow，则可以做到立即响应（告诉用户等着），然后将收到的请求转发给Work线程去处理。WebFlow只会对Work线程形成阻塞，如果再来请求也可以处理。其主要应用场景是：在业务处理很耗时的场景中减少服务器资源的占用，提高并发处理速度。

对于WebFlux的一个简单的理解就是：你来了，我立马答应你，但是服务需要等待；而不是没人理你，咨询服务半天也回复不了。

自动配置

借助AutoConfigurationImportSelector，调用SpringFactoriesLoader的loadFactoryNames方法，从classpath中寻找所有的META-INF/spring.factories配置文件（Spring.factories配置了自动装配的类)，然后，再借助AutoConfigurationImportSelector，将所有符合条件的@Configuration配置都加载到IoC容器中。

SpringBoot功能

  1. 利用@WebFilter来使用Web filter的功能
  2. 利用WebListener来监听资源的的状态变化
  3. 利用spring.factories来实现自定义Starter开发
  4. 利用@Retention源注解来实现创建新的注解
  5. 利用@ControllerAdvice来实现异常统一处理
